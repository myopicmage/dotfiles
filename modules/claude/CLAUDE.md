# Global code rules

These apply to every project, regardless of language or codebase-specific conventions.

- **Match the project's existing conventions first.** Read how the surrounding code is written and follow it — naming, structure, formatting, tooling — even where it differs from my own taste. My defaults, for greenfield code or a project with no established style: **K&R braces** (body on its own line, never a crammed `{ return null; }`), **arrow functions** over the `function` keyword, and **standard linters only** — ESLint for JS/TS with no separate formatter (never Prettier), SwiftLint for Swift. These are defaults, not mandates; the project's own conventions win.
- **Never nest ternaries.** A ternary whose consequent or alternate is itself a ternary must be rewritten — use an if/else chain, a switch, or extract a named function/variable instead. This applies even if the nested ternary is parenthesized or formatted to look flat.
- **Readability beats dependency minimalism.** Prefer clean, readable production code over a shorter dependency list. When a testing or tooling choice pits a dependency against noisier production code — e.g. splitting a function or adding test-only exports purely to avoid a package — keep production clean and take the dependency, especially a dev-only one that isn't shipped. **This is never license to add a dependency silently:** whenever applying this rule (or otherwise) means adding one, call it out explicitly first — its name, whether it's dev-only or shipped to production, and its transitive weight — so it can be vetoed before it lands. Flag heavy or security-relevant ones especially clearly.

# Version control

- **Never squash-merge and never rebase.** Integrate branches with ordinary merge commits (or a fast-forward when the branch is already current) so the full commit history is always preserved. This bans `git rebase`, `git pull --rebase`, interactive rebase, and squash-on-merge. Before stating how a branch merged, verify it — `git log <base>..<branch>` returning 0 commits means the work landed on the base as a real merge/fast-forward — and never loosely call a merge a "squash." If a particular project genuinely *requires* squashing (a team or employer mandate), that project's own instructions override this; default to merge commits everywhere else.
- **Ask about merge policy on a new project — don't infer it.** Signals like the git remote's owner don't reveal it: a repo under a personal account can still be a work project with a mandated merge style, and there's no reliable tell. So on any project whose merge convention isn't already established, just ask how it wants branches integrated (merge commits / squash / rebase) before the first PR or merge, then record the answer (in that project's memory or `CLAUDE.md`) so it's a one-time question. Asking beats guessing here; one question is a cheap, reliable price. When the answer is "your call," the no-squash default above applies.
- **Commit at logical stopping points; ask before pushing or opening a PR.** When the work reaches a natural stopping point, commit it yourself — don't wait for an explicit "commit it." But check with me before pushing a branch or opening a PR, and leave merges to me: local commits are yours to make, anything that leaves this machine is my call.

# Acting on what I say

- **A statement is information, not a task.** An observation, a piece of context, or "huh, that's a lot of X" is me telling you something — acknowledge or discuss it; don't turn it into an action. Acting requires an actual request: an imperative or a clear ask ("commit it," "add the note," "remove that"). Do not infer a task from a comment.
- **When it's ambiguous whether I want action, ask first** — a one-line "want me to X?" — especially before anything consequential (editing config, changing or deleting code, anything outward-facing). Don't overcorrect into asking about everything, though: unambiguous requests just get done. It's the observation-vs-directive line that matters.
- **I often think out loud — I reach an idea *through* discussion, not before it.** In exploratory conversation, be a sounding board: engage, offer angles, push back, and let the idea settle — don't race to implement, and don't force it toward a conclusion or a deliverable. Offering ideas and suggestions is welcome (offer freely); the line is offering vs. quietly acting on a guess.

# Working style

- **Don't generate emails.** I write and send my own — don't produce email drafts or copy-paste email text, and don't offer to.
- **Expect a nix dev shell.** I run nix on every machine I control, so assume a flake is present and use `nix develop 'path:.'` (the `path:.` form avoids the git-tree resolution issue). Any project- or machine-specific nix quirks belong in that project's own notes, not here.
- **I run auth, login, and deploy steps myself.** Prepare the exact commands or a runbook, but don't execute credential, login, or deploy steps on my behalf — I'll run the interactive parts and paste back output.
- **Flag security / access-widening changes for my sign-off.** Any change that loosens a security boundary — weakened auth, a new public read path, a relaxed rule — gets surfaced before commit, even if you think it's low-risk. Raising it is yours; accepting the tradeoff is mine. Don't silently make the call or skip it, and don't recharacterize a correct flag as over-flagging if I accept the risk.
- **Never leave a "needs a build/compile to verify" caveat.** Actually build and verify before calling something done; if the build fails, treat the environment as the thing to fix, not a reason to stop.
- **On a regression, suspect recent version/dependency changes early.** When something that used to work now fails, treat a recent dependency/tool/SDK bump as a first-class suspect — `git log` the lockfiles/manifests (`Package.resolved`, `package-lock.json`, `flake.lock`, `Podfile.lock`, `*.csproj`), compare installed-vs-expected versions, and read the suspect's changelog — *before* assuming the code logic changed. Bisecting the dependency (pin the prior version) is a fast confirm. Cheap to check, and it catches the class of bug where the code is untouched but its foundation moved.
- **Default to correctness over speed.** When the two trade off, take the time to get it right — find the actual root cause, verify it, and don't ship a plausible-looking fix or paper over a failing check just to close faster. A failing test is a signal to investigate, not an obstacle to route around. If I need something quick I'll flag it explicitly; absent that flag, assume I'd rather wait for correct.
- **The `result:` line is for real deliverables.** In background/headless sessions, use it for shipped work, a concrete finding, or a decision — and omit it on conversational, explanatory, or greeting replies.
